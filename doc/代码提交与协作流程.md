# 完整流程（从开始到合并）

1. **更新主分支**
   - 拉取最新代码，确保基于最新主分支开始开发。

2. **创建功能分支**
   - 按 `doc/git规范.md` 命名新分支。

3. **开发与自测**
   - 完成功能开发与必要的本地验证。

4. **本地质量检查与修复（必做）**
   - 推荐先执行 `pnpm fix` 一键修复大部分 Lint 与格式问题。
   - 下面三条都是格式检查
   - `pnpm lint`
   - `pnpm typecheck`
   - `pnpm format:check`

5. **提交前自动修复（由 husky + lint-staged 执行）**
   - 自动执行 ESLint/Stylelint/Prettier 修复。
   - 若失败必须修复后再提交。

6. **提交代码**
   - 使用 `pnpm cz` 生成符合 Conventional Commits 的提交信息。
   - 保持 scope 与实际变更模块一致。

7. **合并前复核**
   - 再次确认 lint/typecheck/format 通过。
   - 为大改动补充变更说明与测试说明。

8. **合并策略**
   - 优先使用 squash 合并，保持历史清晰。

## 分支策略

**强制（必须）**

- 新需求或修复必须在独立分支完成。
- 分支命名遵循 `doc/git规范.md`。

**建议（推荐）**

- 合并前进行自测并完成本地检查。

## 提交流程

**强制（必须）**

- 提交信息必须符合 Conventional Commits。
- 使用 `pnpm cz` 进行交互式提交。

**建议（推荐）**

- 提交范围（scope）保持与实际变更模块一致。

## 合并与评审

**强制（必须）**

- 合并前必须通过 lint/typecheck/format 检查。

**建议（推荐）**

- 大改动需附上变更说明与测试说明。
- 合并方式优先使用 squash，保持历史清晰。

## 冲突处理

**强制（必须）**

- 解决冲突后必须重新运行本地检查。

**建议（推荐）**

- 复杂冲突请记录原因，避免重复冲突。
