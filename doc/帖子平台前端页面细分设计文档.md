# Sea-TryGo 六大业务模块页面规划与设计文档（覆盖版）

> 版本：v2.0  
> 日期：2026-03-15  
> 适用范围：User / Article / Comment / Like / Task / Admin 六大业务模块  
> 落地策略：覆盖现有文档、用户闭环优先、增量兼容路由

---

## A. 现状审计（基于当前代码）

### A.1 已实现页面与路由

| 路由 | 状态 | 说明 |
| --- | --- | --- |
| `/` | 已实现 | 欢迎页，使用 `Layout + PageContainer` |
| `/login` | 已实现 | 登录/注册动效页，已接入 `register/login` |
| `/post` | 已实现 | 发布页，当前走本地 markdown 发布流程 |
| `/dashboard` | 已实现 | 自动重定向到默认 tab |
| `/dashboard/[tab]` | 部分实现 | 仅 `recommend` tab 显示卡片列表，其余 tab 占位 |

### A.2 页面与导航缺口

- 侧边导航存在 `/messages`、`/profile` 入口，但对应页面未实现。
- 缺少文章详情页、个人中心页、任务页、点赞页、完整管理端页面。
- `dashboard` 当前数据来源为本地 `/api/posts`，尚未迁移到文章/推荐接口。

### A.3 BFF 代理现状

当前仓库已具备以下同源代理：

- `/api/usercenter/v1/*` -> `USER_CENTER_API_SERVER_URL`
- `/api/admincenter/v1/*` -> `ADMIN_CENTER_API_SERVER_URL`
- `/api/taskcenter/v1/*` -> `TASK_CENTER_API_SERVER_URL`
- `/api/article/*` -> `ARTICLE_API_SERVER_URL`
- `/api/comment/*` -> `COMMENT_API_SERVER_URL`
- `/api/like/*` -> `LIKE_API_SERVER_URL`
- `/api/reco/*` -> `RECO_API_SERVER_URL`

### A.4 服务层现状

已存在业务服务封装：`auth.ts`、`article.ts`、`comment.ts`、`like.ts`、`task.ts`、`admin.ts`、`reco.ts`。  
已存在统一请求层：`request.ts`（当前默认按 `{ code, msg, data }` 解包）。

### A.5 关键技术缺口清单

1. 缺少用户态与管理态 token 隔离机制（当前仅用户 token key）。
2. 缺少统一路由守卫（未登录与越权访问控制未成体系）。
3. 页面层对接口真实响应结构的兼容性不足（部分接口返回可能非统一包裹）。
4. 文章主流程尚未打通“列表 -> 详情 -> 点赞 -> 评论查看 -> 个人页”闭环。

## B. 六模块子代理规划（汇总版）

> 说明：以下六个“子代理任务卡”分别给出各模块页面规划，最后由本文统一汇总执行。

### B.1 User 子代理任务卡

**当前状态**
- 已完成 `/login` 的注册与登录能力。

**目标页面**
- `/profile`：用户资料页（查看、更新、登出）。
- `/profile/security`：安全设置页（注销账号等高风险操作）。

**接口映射**
- `POST /usercenter/v1/user/register`
- `POST /usercenter/v1/user/login`
- `GET /usercenter/v1/user/get`
- `POST /usercenter/v1/user/update`
- `POST /usercenter/v1/user/logout`
- `DELETE /usercenter/v1/user/delete`

**关键交互**
- 登录成功后写入用户 token 并跳转内容主入口。
- 资料更新后局部刷新用户信息并提示成功。
- 注销账号需二次确认，成功后清理本地 token 并回到登录页。

### B.2 Article 子代理任务卡

**当前状态**
- 已有 `/post` 发布页，但流程仍基于本地 markdown，不是文章服务主链路。
- `/dashboard/recommend` 仍读取本地 `/api/posts`。

**目标页面**
- `/dashboard/recommend`：迁移为文章/推荐数据源。
- `/article/[id]`：文章详情页。
- 编辑能力页（建议增量路由：`/post/edit/[id]`，保留 `/post` 不变）。

**接口映射**
- `GET /v1/articles`
- `GET /v1/article/{id}`
- `POST /v1/article`
- `PUT /v1/article/{id}`
- `DELETE /v1/article/{id}`
- `POST /api/v1/reco/recommend`（推荐增强）

**关键交互**
- 列表优先尝试推荐接口，失败时回退文章列表接口。
- 详情页支持阅读与互动区联动（点赞、评论）。
- 编辑页支持保存、更新、删除并回流详情/列表页。

### B.3 Comment 子代理任务卡

**当前状态**
- 仅有评论列表接口，无新增评论接口。

**目标页面**
- 集成在 `/article/[id]` 页面内的评论区模块。

**接口映射**
- `POST /comment/v1/comment/list`

**关键交互**
- 支持分页加载、空态提示、失败重试。
- 评论输入框保留占位，但提交按钮禁用并明确“暂未开放评论发布”。

### B.4 Like 子代理任务卡

**当前状态**
- 服务层已完整封装 6 个点赞相关接口，页面未接入。

**目标页面**
- `/dashboard/recommend` 列表卡片点赞。
- `/article/[id]` 详情点赞。
- `/profile/likes` 我的点赞与获赞统计页。

**接口映射**
- `POST /like/v1/likeaction`
- `POST /like/v1/getlikestate`
- `POST /like/v1/getlikecount`
- `POST /like/v1/getuserlikelist`
- `POST /like/v1/getusertotallike`
- `POST /like/v1/gettargetlikerlist`

**关键交互**
- 点赞采用乐观更新；失败时回滚并提示。
- 列表页批量查询点赞状态与计数，减少逐条请求。
- 我的点赞页支持游标分页与触底加载。

### B.5 Task 子代理任务卡

**当前状态**
- `task.ts` 与 BFF 代理已就绪，页面未实现。

**目标页面**
- `/profile/tasks` 任务进度页。

**接口映射**
- `POST /taskcenter/v1/task/get`

**关键交互**
- 任务按“进行中/已完成”分组展示。
- 提供手动刷新入口，避免用户误判进度未更新。

### B.6 Admin 子代理任务卡

**当前状态**
- admin 服务封装与 BFF 已就绪，管理端页面未实现。

**目标页面**
- `/admin/login`
- `/admin`
- `/admin/users`
- `/admin/users/[uid]`
- `/admin/profile`

**接口映射**
- `POST /admincenter/v1/admin/login`
- `GET /admincenter/v1/admin/getself`
- `GET /admincenter/v1/admin/getuserlist`
- `GET /admincenter/v1/admin/getuser`
- `POST /admincenter/v1/admin/update`
- `POST /admincenter/v1/admin/updateuser`
- `POST /admincenter/v1/admin/resetuserpassword`
- `POST /admincenter/v1/admin/banuser`
- `POST /admincenter/v1/admin/unbanuser`

- 封禁/解封/删除/重置密码均需操作确认。
- 管理列表支持分页、关键词搜索、详情跳转。

## C. 跨模块基础改造

### C.1 鉴权改造（用户端/管理端隔离）

1. 用户 token 与 admin token 使用不同存储 key。
2. 分离鉴权工具函数：`getUserToken/getAdminToken`、`clearUserToken/clearAdminToken`。
3. 用户页面与管理页面使用不同守卫逻辑和重定向目标。

### C.2 请求层兼容策略

`request.ts` 需要支持两类响应：

1. 包裹响应：`{ code, msg, data }`
2. 直出响应：`{ ... }` 或数组

建议策略：
- 默认兼容解包，保留业务码处理。
- 对确认“直出响应”的接口提供可配置解析模式，避免误判报错。

### C.3 路由守卫策略（增量兼容）

1. 保留现有 `/dashboard`、`/post` 路由入口。
2. 新增受限路由分层：
   - 用户受限：`/profile/*`、部分编辑能力页
   - 管理受限：`/admin/*`
3. 引入统一守卫（推荐 `middleware` + 页面级兜底判断双保险）。

### C.4 页面状态规范

每个新页面必须具备：

1. 加载态：Skeleton 或明确 loading 文案。
2. 空态：说明原因 + CTA。
3. 错误态：错误文案 + 重试按钮。
4. 权限态：未登录/越权的明确跳转或提示。

### C.5 类型收敛策略

1. 保持首期联调可用：不阻塞开发。
2. 对高频页面接口优先收敛强类型：
   - 文章列表/详情
   - 点赞状态/计数
   - 用户资料
3. 低频管理接口分阶段收敛，避免一次性大改。

## D. 分期与优先级（用户闭环优先）

### D.1 P1（最小可用闭环）

目标：登录 -> 推荐列表 -> 文章详情 -> 点赞 -> 评论查看 -> 个人资料

**页面与能力**
- `/dashboard/recommend` 数据源迁移（article/reco）
- `/article/[id]` 详情页 + 点赞 + 评论列表
- `/profile` 用户资料页（查看、更新、登出）

**依赖**
- 请求层兼容策略落地
- 用户 token 守卫

### D.2 P2（个人中心扩展）

目标：提高留存与自服务能力

**页面与能力**
- `/profile/likes` 我的点赞与获赞
- `/profile/tasks` 任务进度
- `/profile/security` 注销等高风险操作

**依赖**
- Like、Task 相关列表分页与状态规范

### D.3 P3（管理端建设）

目标：形成后台运营与风控能力

**页面与能力**
- `/admin/login`
- `/admin`
- `/admin/users`
- `/admin/users/[uid]`
- `/admin/profile`

**依赖**
- admin token 隔离
- 管理端路由守卫
- 高风险操作确认流统一组件

### D.4 增量兼容说明

1. 旧入口保留：`/dashboard`、`/post` 不立即下线。
2. 新页面逐步接入，避免一次性路由重排造成断链。
3. 导航在新页面稳定后再统一改版。

## E. 验收标准（接口联调 + 页面验收）

### E.1 页面验收

1. 现有页面不回归：`/login`、`/dashboard/*`、`/post`。
2. 新增页面可访问，且具备加载态/空态/错误态。
3. 关键跳转链路无死链。

### E.2 六模块最小联调用例

每个模块至少覆盖成功与失败各 1 条：

1. User：登录成功、token 失效获取资料失败。
2. Article：文章列表成功、详情 404 失败。
3. Comment：评论列表成功、分页越界空列表。
4. Like：点赞成功、点赞失败回滚。
5. Task：任务进度成功、接口异常重试。
6. Admin：管理员登录成功、越权访问失败。

### E.3 鉴权验收

1. 用户 token 与 admin token 相互隔离。
2. 未登录访问受限页会被正确重定向。
3. 管理端越权访问不会泄漏页面数据。

### E.4 兼容验收

1. 增量路由策略下旧入口继续可用。
2. 新旧数据链路切换期间页面有兜底展示，不出现白屏。
3. 同源 BFF 不可用时前端可识别 502 并提示重试。

### E.5 质量基线

实施阶段每次提交需通过：

- `pnpm typecheck`
- `pnpm lint:eslint`

## 附录：本轮范围边界

1. `/messages` 仍为占位入口，不纳入本轮六模块实现范围。
2. Comment 模块仅支持“评论列表查看”，发布评论能力等待后端接口开放后再接入。
3. 若后端返回结构与文档不一致，以联调抓包结果为准并同步更新本设计文档。
