### 2.2 当前前端仓库可用 BFF 代理

| 前端同源前缀 | 转发目标环境变量 | 说明 |
| --- | --- | --- |
| `/api/usercenter/v1/*` | `USER_CENTER_API_SERVER_URL` | 用户中心专用代理 |
| `/api/article/*` | `ARTICLE_API_SERVER_URL` | 透传到文章服务根路径 |
| `/api/comment/*` | `COMMENT_API_SERVER_URL` | 透传到评论服务根路径 |
| `/api/like/*` | `LIKE_API_SERVER_URL` | 透传到点赞服务根路径 |

- 当前仓库未提供 admin/task 的同源代理路由。

### 2.3 鉴权规范

- `bearer` / `jwt` 均为 HTTP Bearer Token 方案。
- 统一请求头：`Authorization: Bearer <token>`。
- Token 来源：
  - 用户端：`POST /usercenter/v1/user/login`
  - 管理端：`POST /admincenter/v1/admin/login`

### 2.4 请求/响应规范

- 请求 `Content-Type`：推荐 `application/json`。
- 前端现有 `src/services/request.ts` 默认按 `{ code, msg, data }` 包装解析。
- ⚠️ 不确定因素：MCP 导出的部分接口响应 schema 为空对象或未给示例，实际返回体可能因服务实现存在差异。
  - 建议：联调时先抓包确认真实字段，再在前端类型中固化。

### 2.5 Article 路径特别说明

- MCP 导出中出现固定 ID 路径：
  - `/v1/article/2029849189820395520`（DELETE）
  - `/v1/article/2029851019848585216`（GET）
  - `/v1/article/2029855154559172608`（PUT）
- 基于 REST 语义，实际对接建议按 `/{id}` 动态替换，即 `/v1/article/{id}`。

## 4. 推荐联调规范

### 4.1 必做校验

- 先验证 token 生效：登录后立即调用一个鉴权接口（如 `user/get`、`admin/getself`）。
- 对空 schema 接口，必须在联调阶段落地 TS 类型定义。
- 对列表类接口（`getuserlist`、`articles`、like/comment 分页）统一验证：
  - 空列表
  - 大分页
  - 末页/游标结束

### 4.2 错误处理建议

- HTTP 非 2xx：按 `status + body` 兜底提示。
- 业务码非 200（若存在 `code` 字段）：按 `msg` 透出。
- 代理不可用（BFF）：当前项目会返回 `502` 和 `{ code: 502, msg: "xxx服务不可用", data: null }`。

### 4.3 前端调用建议

- 浏览器侧优先走同源 BFF（已存在 user/article/comment/like），减少跨域和暴露内网地址。
- admin/task 若需浏览器调用，建议补充同源代理路由后再接入页面。
- 对文章接口的 `{id}` 路径建议在调用层统一封装，避免硬编码。
